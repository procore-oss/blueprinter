Blueprinter

Views

Blueprints can define views to provide different representations of the data. A view inherits everything from its parent context but is free to override as needed. In addition to fields, views can define options, partials, formatters, extensions, and nested views.

At the top level of every Blueprint is an implicit view called default. This view is used when no other is specified.

class WidgetBlueprint < ApplicationBlueprint
  field :name
  object :category, CategoryBlueprint

  # The "with_parts" view inherits from "default" and adds a collection of parts
  view :with_parts do
    collection :parts, PartBlueprint
  end

  # Views can include other views
  view :full do
    use :with_parts
    field :description
  end
end

Excluding fields

Views can exclude fields from a parent:

class WidgetBlueprint < ApplicationBlueprint
  fields :name, :description, :price

  view :minimal do
    exclude :description, :price
  end
end

Nested views

You can nest views within views, allowing for a hierarchy of inheritance.

class WidgetBlueprint < ApplicationBlueprint
  field :name
  object :category, CategoryBlueprint

  view :extended do
    field :description
    collection :parts, PartBlueprint

    # The "extended.with_price" view adds a price field
    view :with_price do
      field :price
    end
  end

Referencing views

When defining an association, you can choose a view from its blueprint:

object :widget, WidgetBlueprint[:extended]
collection :widgets, WidgetBlueprint["extended.with_price"]

Inheriting from views

You can inherit from another blueprint's view:

class WidgetBlueprint < ApplicationBlueprint[:with_timestamps]
  # ...
end
